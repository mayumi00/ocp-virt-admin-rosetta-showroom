
= Decentralized Live Migration

ifdef::vmware[]
.Rosetta Stone: VMware
****
OpenShiftは、VMwareと同様にストレージとコンピュートのライブマイグレーションを行います。

VMwareにおけるストレージマイグレーションはすでに組み込まれています。

この機能は、Cross-vCenter vMotion と同様の機能およびメリットを提供します。
****
endif::[]

分散型ライブマイグレーション（Decentralized Live Migration）は、ストレージライブマイグレーションのバリエーションの1つであり、実行中の *VirtualMachine* を *ネームスペース* 間や *クラスター* 間でマイグレーションすることができます。

これは以下のような場合に役立ちます。

*ワークロードのバランシング（負荷分散）*：クラスター間で仮想マシンを分散させることで、リソースの利用率を最適化できます。1つのクラスターの負荷が非常に高く、もう1つのクラスターがアイドル状態である場合、再配置を行うことで運用の効率性を大幅に向上させることができます。

*メンテナンスの円滑化*：複数のクラスターが存在する環境では、仮想マシンのマイグレーションによってシームレスなメンテナンスが可能になります。アップグレードやシャットダウンが予定されているクラスターから仮想マシンを移動させることで、サービスのダウンタイムをゼロに抑えることができます。

*迅速なリストア（復元）*：バックアップベンダーが提供するインスタントリストア機能は、特にネームスペースのマイグレーションと組み合わせることで、復旧時間を劇的に短縮できます。仮想マシンを一時的な場所に素早くリストアした後、元のネームスペースとストレージにマイグレーションすることができます。

このラボでは、今後登場するテックプレビュー（Tech Preview）機能である「分散型ライブマイグレーション」の機能を実証します。このラボでは、ある *Namespace（ネームスペース）* から別のネームスペースへの *VirtualMachine* のライブマイグレーション手順を実行します。ネームスペース間でライブマイグレーションを行うプロセスは、2つのクラスター間で *VirtualMachine* をマイグレーションする場合とまったく同じプロセスに従います。

[NOTE]
====
UIベースのワークフローはまだ完全には機能していません。このラボでは、UIでのワークフローを紹介しますが、実際のマイグレーションはCLIを通じて実行します。
====

[#howdeclm]
== How does it work?

このマイグレーションには、2つの *VirtualMachineInstance* と2つの *VirtualMachineInstanceMigration* オブジェクトが関与します。ストレージライブマイグレーションと同様に、ディスクの内容はネットワーク経由で受信側の *VirtualMachine* にコピーされます。主な違いは、受信側の仮想マシンが完全に別の *VirtualMachineInstance* を持っている点です。マイグレーションを調整するために、移行元（ソース）と移行先（ターゲット）の *VirtualMachineInstance* のステータスを同期させる必要があります。*openshift-cnv* ネームスペースで実行される専用の同期コントローラー（`virt-synchronization-controller`）が、ソースとターゲットの *VirtualMachineInstance* 間の通信を仲介します。

[#decmlrequirements]
== Requirements

これらの要件はラボのクラスターであらかじめ有効化されており、参照用に記載しています。

* *Migration Toolkit for Virtualization (MTV)* Operator がインストールされており、*ForkliftController* CRを作成する際に機能フラグ *feature_ocp_live_migration* が *true* に設定されていること。
* *hyperconverged* カスタムリソースで *DecentralizedLiveMigration* の *featureGate* を有効にしていること。

[#decmlinstructions]
== CLI Based Instructions

. *Webブラウザ* からOpenShiftコンソールに、画面右側の *ターミナル* ウィンドウからCLIに、それぞれ *admin* ユーザーとしてログインしていることを確認し、次のステップに進みます。
+

. 移行元（ソース）の仮想マシンを起動します。

+
[.wrap,console,role=execute]
----
virtctl start vm-migration-ns-ns-live -n vm-live-migration-source
----

+
[source,console]
----
VM vm-migration-ns-ns-live was scheduled to start
----
+
. ソースの *VirtualMachineInstance* が実行中であることを確認します。

+
[.wrap,console,role=execute]
----
oc get vmi -n vm-live-migration-source
----

+
出力は以下のようになります（*IP* や *NODENAME* は環境によって異なります）。

+
[source,console]
----
NAME                     AGE  PHASE    IP          NODENAME               READY
vm-migration-ns-ns-live  28m  Running  10.233.2.16 worker-cluster-hflz6-3 True
----
+
それでは、仮想マシン *vm-migration-ns-ns-live* を、ネームスペース *vm-live-migration-source* から新しいネームスペース *vm-live-migration-destination* へマイグレーションしてみましょう。

[start=3]
. 先述の通り、このマイグレーションには2つの *VirtualMachineInstance* が必要です。
+
最初のステップとして、移行先（デスティネーション）のネームスペースに、受信側仮想マシン用の空の *DataVolume* を作成します。
+
次のコマンドを実行して、移行先の *DataVolume* を作成します。

+
[.wrap,yaml,role=execute]
----
cat <<EOF | oc apply -f -
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  annotations:
    cdi.kubevirt.io/storage.usePopulator: "true"
  name: vm-migration-ns-ns-live
  namespace: vm-live-migration-destination
spec:
  source:
    blank: {}
  storage:
    storageClassName: ocs-external-storagecluster-ceph-rbd
    resources:
      requests:
        storage: 30Gi
EOF
----
+
次の2つのコマンドを実行して、*DataVolume* とそれに関連付けられた *PersistentVolumeClaim* が作成されたことを確認します。
+
マイグレーションが開始されるまで、DV と PVC はそれぞれ *PendingPopulation* および *Pending* 状態になるのが正常な挙動です。

+
[.wrap,console,role=execute]
----
oc get DataVolume -n vm-live-migration-destination
----

+
[source,console]
----
NAME                     PHASE              PROGRESS  RESTARTS  AGE
vm-migration-ns-ns-live  PendingPopulation  N/A                 107s
----

+
[.wrap,console,role=execute]
----
oc get PersistentVolumeClaim -n vm-live-migration-destination
----

+
[source,console]
----
NAME                     STATUS   VOLUME  CAPACITY  ACCESS MODES  STORAGECLASS                          VOLUMEATTRIBUTESCLASS  AGE
vm-migration-ns-ns-live  Pending                                  ocs-external-storagecluster-ceph-rbd                  2m5s
----

+
[start=4]
. 次に、移行先のネームスペースに受信側の仮想マシンを作成します。
+
移行先の *VirtualMachine* は、以下の2つの重要な相違点を除き、移行元の *VirtualMachine* と同じです。
+
[loweralpha]
.. 移行先の *VirtualMachine* には、ライブマイグレーション後の *runStrategy* を設定するためのアノテーション（注釈）が付与されています。

+
[source,yaml]

----
    kubevirt.io/restore-run-strategy: Always
----

+
[loweralpha,start=2]
.. 移行先の仮想マシンは、異なる *spec.runStrategy* を持っています。

+
[source,yaml]
----
  runStrategy: WaitAsReceiver
----

+
. 次のコマンドを実行して、移行先の仮想マシンを作成します。

+
[.wrap,yaml,role=execute]
----
cat <<EOF | oc apply -f -
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  annotations:
    kubevirt.io/restore-run-strategy: Always
  name: vm-migration-ns-ns-live
  namespace: vm-live-migration-destination
spec:
  runStrategy: WaitAsReceiver
  template:
    metadata:
      annotations:
        vm.kubevirt.io/flavor: small
        vm.kubevirt.io/os: rhel9
        vm.kubevirt.io/workload: server
      creationTimestamp: null
      labels:
        kubevirt.io/domain: vm-migration-ns-ns-live
        kubevirt.io/size: small
        network.kubevirt.io/headlessService: headless
    spec:
      architecture: amd64
      networks:
      - name: default
        pod: {}
      domain:
        cpu:
          cores: 1
          sockets: 1
          threads: 1
        devices:
          disks:
          - disk:
              bus: virtio
            name: rootdisk
          - disk:
              bus: virtio
            name: cloudinitdisk
          interfaces:
          - macAddress: 02:a1:3b:00:00:85
            masquerade: {}
            model: virtio
            name: default
          logSerialConsole: false
          rng: {}
        features:
          acpi: {}
          smm:
            enabled: true
        firmware:
          bootloader:
            efi: {}
          serial: 94f58dd1-b1e1-4aab-add2-ab3d3483d297
        machine:
          type: pc-q35-rhel9.6.0
        memory:
          guest: 2Gi
        resources: {}
      terminationGracePeriodSeconds: 180
      volumes:
      - dataVolume:
          name: vm-migration-ns-ns-live
        name: rootdisk
      - cloudInitNoCloud:
          userData: |-
            #cloud-config
            user: cloud-user
            password: redhat
            chpasswd: { expire: False }
        name: cloudinitdisk
EOF
----

+
次の2つのコマンドを実行して、*VirtualMachine* とそれに関連付けられた *VirtualMachineInstance* が作成されたことを確認します。

+
[.wrap,console,role=execute]
----
oc get VirtualMachine -n vm-live-migration-destination
----

+
[source,console]
----
NAME                      AGE     STATUS               READY
vm-migration-ns-ns-live   51s     WaitingForReceiver   False
----

+
[.wrap,console,role=execute]
----
oc get VirtualMachineInstance -n vm-live-migration-destination
----

+
[source,console]
----
NAME                      AGE     PHASE            IP    NODENAME   READY
vm-migration-ns-ns-live   5m55s   WaitingForSync                    False
----
+
*VirtualMachine* と *VirtualMachineInstance* が、それぞれ *WaitingForReceiver* および *WaitingForSync* という固有の *PHASE（フェーズ）* になっていることが確認できます。
+
これは、*VirtualMachine* が受信側（Receiver）としてマイグレーションの開始を待機しており、*VirtualMachineInstance* はソースの *VirtualMachine* からデータが同期されるのを待機している状態を示しています。

+
[start=6]
. 移行先の *VirtualMachine* と *VirtualMachineInstance* が待機状態になったら、次のコマンドを実行して移行先の *VirtualMachineInstanceMigration* を作成します。

+
[.wrap,yaml,role=execute]
----
cat <<EOF | oc apply -f -
apiVersion: kubevirt.io/v1
kind: VirtualMachineInstanceMigration
metadata:
  name: ns-to-ns-vm-live-migration-instance-destination
  namespace: vm-live-migration-destination
spec:
  receive:
    migrationID: 52e4398d-bdbf-42b5-b0f4-1e7c6c0a08f5-38cec1f6-43bb-412d-8477-b3d635fd7123
  vmiName: vm-migration-ns-ns-live
EOF
----

+
[NOTE]
====
上記のYAMLにおける *migrationID* フィールドは特定のオブジェクトを指すものではありませんが、すべてのマイグレーションにおいて一意（ユニーク）である必要があります。このフィールドには任意の一意のテキストを使用できますが、1つのマイグレーションに対してソース側とデスティネーション側の両方の *VirtualMachineInstanceMigration* オブジェクトで同じ *migrationID* を使用する必要があります。
====

+
次のコマンドを実行して、移行先の *VirtualMachineInstanceMigration* が作成されたことを確認します。

+
[.wrap,console,role=execute]
----
oc get VirtualMachineInstanceMigration -n vm-live-migration-destination
----

+
[source,console]
----
NAME                                              PHASE            VMI
ns-to-ns-vm-live-migration-instance-destination   WaitingForSync   vm-migration-ns-ns-live
----

[start=7]
. 移行先の *VirtualMachineInstanceMigration* が待機状態になったら、最後のステップとして、移行元の *VirtualMachineInstanceMigration* を作成します。
+
移行元の *VirtualMachineInstanceMigration* を作成するには、リーダー（アクティブ）である *virt-synchronization-controller* の *IPアドレス* が必要です。
+
[loweralpha]
.. リースを保持している *virt-synchronization-controller* のIPアドレスを見つけます。

+
[.wrap,console,role=execute]
----
oc get pods $(oc get leases virt-synchronization-controller -n openshift-cnv -o json | jq -r '.spec.holderIdentity') -o json -n openshift-cnv | jq -r '.status.podIP'
----

+
[IMPORTANT]
====
お使いのクラスターの *IPアドレスはこれとは異なります*！出力された正しいIPアドレスを使用していることを必ず確認してください！
====

+
.出力例
[source,console]
----
10.232.1.136
----

+
. 前のコマンドの出力から *ポッドのIPアドレス* （例では *10.232.1.136*）をコピーし、以下の *VirtualMachineInstanceMigration* YAMLの *<leader_sync_controller_ip>* の部分を置き換えます。その際、ポート番号の `:9185` を削除してしまわないよう注意してください。

+
[.wrap,yaml,role=execute]
----
cat <<EOF | oc apply -f -
apiVersion: kubevirt.io/v1
kind: VirtualMachineInstanceMigration
metadata:
  name: ns-to-ns-vm-live-migration-instance-source
  namespace: vm-live-migration-source
spec:
  sendTo:
    connectURL: <leader_sync_controller_ip>:9185
    migrationID: 52e4398d-bdbf-42b5-b0f4-1e7c6c0a08f5-38cec1f6-43bb-412d-8477-b3d635fd7123
  vmiName: vm-migration-ns-ns-live
EOF
----

[#decmlmonitor]
== Monitor the Migration
上記の移行元の *VirtualMachineInstanceMigration* が作成されると、マイグレーションが開始されます。マイグレーションの進捗は、CLIまたはコンソールから監視できます。

*Webブラウザ* からOpenShiftコンソールに、画面右側の *ターミナル* ウィンドウからCLIに、それぞれ *admin* ユーザーとしてログインしていることを確認し、次のステップに進みます。

. CLIからの確認方法

+
[.wrap,console,role=execute]
----
oc get vmim -A -w
----

+
[NOTE]
====
`-w` パラメーターを使用して監視（Watch）を適用すると、*Migration* が *Scheduling* から *Succeeded* まで、いくつかの一連の *PHASE* を経て進むのを確認できます。
====

+
[source,console]
----

+
[.wrap,console,role=execute]
----

*VirtualMachine* のステータスが *Migrating（マイグレーション中）* と表示されます。

+

. コンソールからの確認方法

+
左側のサイドパネルから、*Virtualization（仮想化） -> Virtual Machines（仮想マシン）* に移動します。*All projects（すべてのプロジェクト）* の下で、*vm-live-migration-source* ネームスペースを選択し、*vm-migration-ns-ns-live* という名前の仮想マシンをクリックします。
+
*Virtual machine details（仮想マシンの詳細）* ページで、移行元（*Source*）のVMが *Running（実行中）* であることが確認できます。

+
image::exercise7/image1-ui-src-vm-running.png[title="Source Virtual Machine Details - Running", link=self, window=blank, width=100%]

+
移行先（*Destination*）のVMを表示するには、*Namespace* ドロップダウンから *vm-live-migration-destination* を選択して展開し、*vm-migration-ns-ns-live* という名前の仮想マシンをクリックします。
+
*Virtual machine details（仮想マシンの詳細）* ページに、移行先のVMが表示されます。マイグレーションがまだ開始されていない場合、ステータスは *WaitingForReceiver* になります。

image::exercise7/image2-ui-dest-waiting.png[title="Destination Virtual Machine Details - WaitingForReceiver", link=self, window=blank, width=100%]

+
マイグレーションが開始され、初期段階にある場合、VMのステータスは *Starting（起動中）* になります。

+
image::exercise7/image3-ui-dest-starting.png[title="Destination Virtual Machine Details - Starting", link=self, window=blank, width=100%]

+
ソースとデスティネーションの両方の準備が整うと、移行先VMのステータスが *Migrating（マイグレーション中）* に変わります。

image::exercise7/image4-ui-dest-migrating.png[title="Destination Virtual Machine Details - Migrating", link=self, window=blank, width=100%]

移行先VMのステータスが *Migrating* に変わったら、その *Migrating* ステータスをクリックすることで、マイグレーションの状態に関する情報を記載したポップアップパネルを表示できます。

*Migrating* ポップアップパネル内に、*Migration metrics（マイグレーションメトリクス）* というリンクがあります。そのリンクをクリックすると、VMとマイグレーションに関する詳細情報が含まれる *Metrics（メトリクス）* タブに移動します。
+
image::exercise7/image6-ui-dest-migrating-metrics.png[title="Destination Virtual Machine - Migration Metrics", link=self, window=blank, width=100%]
+
*Metrics* ページを下にスクロールすると、さらなるマイグレーションメトリクスと *LiveMigration progress（ライブマイグレーションの進捗）* バーが表示されます。
+
image::exercise7/image7-ui-dest-migrating-metrics2.png[title="Destination Virtual Machine - Migration Metrics", link=self, window=blank, width=100%]
+
マイグレーションが進行するにつれて、*LiveMigration progress* バーが動き、*100%* に達すると *Complete time（完了時間）* が表示されます。
+
image::exercise7/image8-ui-dest-migrating-progress-bar.png[title="Destination Virtual Machine - Migration Progress Bar", link=self, window=blank, width=100%]
+
*Overview（概要）* タブに戻ると、移行先（*Destination*）のVMが *Running（実行中）* 状態になっていることが確認できます。また、サイドバーの *Projects* を見ると、移行元（*Source*）のVMが現在 *Stopped（停止）* になっていることも確認できます。
+
これにより、ネームスペースからネームスペースへのライブマイグレーションが正常に完了したことになります。

[IMPORTANT]

. *Webブラウザ* からOpenShiftコンソールに *admin* ユーザーとしてログインしていることを確認し、次のステップに進みます。

----
+
. 左側のサイドパネルから、*Migration for Virtualization（仮想化のマイグレーション）* に移動し、*Migration plans（マイグレーションプラン）* をクリックします。
+
image::exercise7/ui/image1-click-migration-plans.png[title="Migration Plans", link=self, window=blank, width=100%]

+
. *Migration plans* ページから、*Create plan（プランの作成）* をクリックします。

+
[NOTE]
====
*Create plan* ボタンがグレーアウトしている場合は、ページ上部の *Project:* ドロップダウンから、プロジェクトが *openshift-mtv* または *All projects（すべてのプロジェクト）* に選択されていることを確認してください。
====
+
image::exercise7/ui/image2-click-create-plans.png[title="Create Migration Plan", link=self, window=blank, width=100%]

+
. *Creation migration plan（マイグレーションプランの作成）* ページで、プランに `namespace-to-namespace` のような名前を入力します。
+
*Plan project（プランプロジェクト）* はデフォルト値の *openshift-mtv* のままにします。

+
image::exercise7/ui/image3-planinfo-name-project.png[title="Create migration plan", link=self, window=blank, width=100%]


. 同じ画面のさらに下部で、*Target（ターゲット）* と *Source（ソース）* のプロバイダーの両方に *host* を選択し、*Target project（ターゲットプロジェクト）* として *vm-live-migration-destination* を選択します。
+
*Next（次へ）* をクリックします。
+
[NOTE]
====
今回は同じクラスター内の *Namespaces* 間で *VirtualMachine* をマイグレーションするため、*Source* と *Target* のプロバイダーは同じになります。もしクラスター間でマイグレーションを行う場合は、*Target* クラスターに対して異なるプロバイダーが存在するため、代わりにそれを選択することになります。
====

+
image::exercise7/ui/image4-planinfo-source-target-providers.png[title="Create migration plan", link=self, window=blank, width=100%]

. *Virtual machines* のリストから、マイグレーションしたい仮想マシンとして *vm-migration-ns-ns-live* を選択します。
+
*Next（次へ）* をクリックします。

+
image::exercise7/ui/image5-select-virtual-machine.png[title="Select Virtual Machine", link=self, window=blank, width=100%]

. *Network Map（ネットワークマップ）* ページで、*Use new network map（新しいネットワークマップの使用）* を選択し、*Source network（ソースネットワーク）* として */Default network* を選択します。
+
*Target network（ターゲットネットワーク）* はすでに *Default network* に設定されているはずです。
+
*Next（次へ）* をクリックします。

+
image::exercise7/ui/image6-new-network-map.png[title="Create Network Map", link=self, window=blank, width=100%]

. *Storage Map（ストレージマップ）* ページで、*Use new storage map（新しいストレージマップの使用）* を選択し、*Source storage（ソースストレージ）* として *ocs-external-storagecluster-ceph-rbd* を選択します。
+
*Target storage（ターゲットストレージ）* はすでに *ocs-external-storagecluster-ceph-rbd* に設定されているはずです。
+
*Next（次へ）* をクリックします。

+
image::exercise7/ui/image7-new-storage-map.png[title="Create Storage Map", link=self, window=blank, width=100%]

. *Migration type（マイグレーションタイプ）* ページで、*Live migration（ライブマイグレーション）* を選択します。
+
*Next（次へ）* をクリックします。

+
image::exercise7/ui/image8-migration-type-live.png[title="Migration Type", link=self, window=blank, width=100%]

. *Other settings（その他の設定）* ページは、デフォルトのままにします。
+
*Next（次へ）* をクリックします。

+
image::exercise7/ui/image9-power-state.png[title="Other Settings", link=self, window=blank, width=100%]

. *Hooks（フック）* ページは、デフォルトのままにします。
+
*Next（次へ）* をクリックします。

+
image::exercise7/ui/image10-migration-hooks.png[title="Hooks", link=self, window=blank, width=100%]

. *Review and create（確認と作成）* ページで、すべての内容が正しいことを確認します。
+
*Create plan（プランの作成）* をクリックします。

+
image::exercise7/ui/image11-review-create.png[title="Review and Create", link=self, window=blank, width=100%]

. *Plan* が作成されると、マイグレーションのステータスと進捗を監視できる *Plan details（プランの詳細）* ページに移動します。
+
[NOTE]
====
画面に *The plan is not ready* と表示され、*MACアドレスの競合（MAC address conflict）* が原因でステータスが *Cannot start（開始不可）* になっていることが確認できます。

この問題が解決されれば、ワークフローを変更することなく、マイグレーションは正常に続行されます。

詳細については、バージョン 4.22 をターゲットとして作業が進められている https://issues.redhat.com/browse/CNV-72966[CNV-72966 - loosen (or drop) mac collision detection]をフォローして確認することができます。
====

+
image::exercise7/ui/image12-plan-created-error.png[title="Plan Details", link=self, window=blank, width=100%]

[IMPORTANT]
====
次のラボ用に十分なリソースを確保するため、*Terminal（ターミナル）* ウィンドウから *virtctl* を使用して、移行先ネームスペースにある *vm-migration-ns-ns-live* VM を停止してください。

[,shell,role="execute wrap",subs="attributes"]
----
virtctl stop vm-migration-ns-ns-live -n vm-live-migration-destination
----

[source,console]
----
VM vm-migration-ns-ns-live was scheduled to stop
----
====
