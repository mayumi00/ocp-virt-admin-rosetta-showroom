*オーバーコミット（Overcommit）* は、割り当てられた仮想リソースがホスト上で利用可能な物理リソースを超えたときに発生します。仮想マシン（VM）が割り当てられた最大容量を同時にフルで使用することは滅多にないという事実を活用することで、オーバーコミットによりワークロードの収容密度を向上させることができます。

このモジュールでは、2つのオーバーコミットタスクを実行します。

* <<memory,仮想マシンにおけるメモリオーバーコミットの理解、有効化、および設定>>

[#cpu]
== CPU Overcommit

OpenShift Virtualizationにおいて、仮想マシン（VM）に割り当てられるコンピュートリソースは、*Guaranteed CPU（保証型CPU）* または *タイムスライス型CPUシェア* のいずれかによって支えられています。

*Guaranteed CPU* は、CPUリザベーションとも呼ばれ、特定のワークロードにCPUコアまたはスレッドを専用に割り当て、他のワークロードが使用できないようにします。VMにGuaranteed CPUを割り当てると、予約された物理CPUへの排他的なアクセスが保証されます。VMでGuaranteed CPUを使用するには、専用リソース（Dedicated Resources）を有効にします。

*タイムスライス型CPU* は、共有された物理CPU上の時間の断片（スライス）を各ワークロードに割り当てます。このスライスのサイズは、VMの作成時またはVMのオフライン時に指定できます。デフォルトでは、各vCPUは物理CPU時間の100ミリ秒（1/10秒）を受け取ります。

タイムスライス型CPUを使用する場合、Linuxカーネルの *Completely Fair Scheduler (CFS)* が、VM間で物理CPUコアをどのように共有するかを管理します。CFSは利用可能なコア間でVMをローテーションさせ、構成されたCPU要求（Requests）に基づいて、各VMに比例したCPU時間スライスを割り当てます。

デフォルトで、OpenShift Virtualizationは10:1のオーバーコミット比率に設定されています。CPUオーバーコミットを実現するため、VMが要求するvCPUごとに、Kubernetesのリソースおよびスケジューリングの観点から、各VMの *virt-launcher* ポッドは *100m*（CPUの1/10）のCPU要求（Requests）を定義します。

デフォルトの *10:1* のオーバーコミット比率は、*hyperconverged* カスタムリソース（CR）の *vmiCPUAllocationRatio* を変更することで、希望するオーバーコミットレベルに構成できます。この比率を変更すると、デフォルトで割り当てられる各vCPUのCPU要求（Requests）に影響を与え、Kubernetesベースの要求スケジューリングを通じてCPUオーバーコミットの最大レベルが強制されます。

[NOTE]
====
リソースの割り当ては *virt-launcher* ポッドのスケジューリング時に行われるため、比率の変更後にCPU割り当て動作を反映させるには、VMをライブマイグレーションするか、停止して再起動する必要があります。
====

[#memory]
== Memory Overcommit

Kubernetes（1.21まで）およびOpenShift（4.21まで）には、一般利用可能（GA）なネイティブのSWAP実装がなかったため、SWAPの使用が許可されていませんでした。

スワップを使用しないメモリのオーバーサブスクリプション（過剰割り当て）は危険です。ノード上で実行されているプロセスが必要とするメモリ量が利用可能なRAMの量を超えると、プロセスが強制終了（Kill）されるためです。これは特に、VMが強制終了されるとワークロードがオフラインになってしまう仮想マシンの環境において望ましくありません。

OpenShift Virtualizationは、VMでのスワップの制御された使用を許可するために *wasp-agent* を同梱しています。
詳細については、 https://docs.redhat.com/en/documentation/openshift_container_platform/4.20/html/virtualization/postinstallation-configuration#virt-using-wasp-agent-to-configure-higher-vm-workload-density_virt-configuring-higher-vm-workload-density[wasp-agent component documentation]  を参照してください。

NOTE: 時間とリソースの制約のため、このラボでは *wasp-agent* の構成は行いませんが、各自でドキュメントやインストール手順を確認することができます。

IMPORTANT: Kubernetes 1.21 および OpenShift 4.21 以降、*wasp-agent* はネイティブの Kubernetes SWAP 実装に置き換わる予定です。

*hyperconverged* カスタムリソースの *memoryOvercommitPercentage* パラメーターは、各VMのメモリ要求（Requests）をどのようにスケールするかを OpenShift Virtualization に指示します。デフォルトの 100 に設定されている場合、VMによって宣言された完全なメモリ量に基づいてメモリ要求を計算します。これより高い値に設定すると、要求（Request）はVMが要求した量よりも比例して小さい値に設定され、メモリのオーバーコミットが可能になります。

例えば、16GiBのメモリを持つVMの場合：

オーバーコミットのパーセンテージがデフォルト値の 100 に設定されている場合、ポッド上のメモリ要求は 16 GiB に、VMを実行する QEMU プロセスのオーバーヘッド（割当余剰分）を加えたものになります。

これが 200 に設定されている場合、ポッド上の要求は 8GiB にオーバーヘッドを加えたものに設定されますが、VM内部からは引き続き 16GiB として認識されます。

[IMPORTANT]
====
メモリをオーバーコミットする場合の要求値（Request）は、次の数式を使って計算できます。

要求されるメモリ = VMのメモリ * (100 / memoryOvercommitPercentage)

16 * (100 / 200) = 8GiB

または、150% のオーバーコミットの場合：

16 * (100 / 150) = 10.66GiB
====

[NOTE]
====
仮想マシンあたりのメモリオーバーヘッド ≈ (0.002 × 要求されるメモリ) + 218 MiB + 8 MiB × (vCPUの数) + 16 MiB × (グラフィックスデバイスの数) + (追加のメモリオーバーヘッド)

OpenShift Virtualization ドキュメントの https://docs.redhat.com/en/documentation/openshift_container_platform/4.20/html/virtualization/installing#memory-overhead_preparing-cluster-for-virt[仮想マシンのメモリオーバーヘッド（Virtual machine memory overhead）] のセクションを参照してください。
====
