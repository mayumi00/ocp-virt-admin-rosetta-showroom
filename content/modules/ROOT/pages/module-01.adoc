
= Live Migration of Workloads

ifdef::vmware[]
.Rosetta Stone: VMware
****
VMwareでは、これは「vMotion」と呼ばれます。

VMwareは「データストア（Datastores）」の概念を使用します。
VMwareのデータストアはLUN（Logical Unit Number）と考えることができ、データストアクラスターに属している場合とそうでない場合があります。

.Differences

* パラダイムが異なります。
* PV（PersistentVolume）や PVC（PersistentVolumeClaim）は、VMが必要としたときに StorageClass から取得して提供します。
* データストア -> クラスターという構造です。

.Mapping

* データストアクラスター（またはデータストアに直接接続されたVM） = StorageClass がボリュームを定義し、それに対して要求（Claims）を行います。

データストアクラスターはデータストアの集合体であり、特定のデータストアへの `.VMDK` ファイルの配置に SDRS を使用します。
Storage Distributed Resource Scheduler (SDRS) はデータストアクラスターにのみ適用可能で、このプロセスを自動化するために使用されます。
****
endif::[]

コンピュートノード間やストレージタイプ間でワークロードを移行することは、現代のあらゆるハイパーバイザーにおいて極めて重要な機能です。OpenShift Virtualizationでは、仮想マシンのコンピュートとストレージをある場所から別の場所へ、シンプルかつ簡単に移行できます。

このモジュールでは、3つのマイグレーションタスクを実行します。

* <<computerand,任意のノードへの仮想マシンのライブマイグレーション>>
* <<compute,特定のノードへの仮想マシンのライブマイグレーション>>
* <<storage,ある StorageClass から別の StorageClass への仮想マシンストレージのライブマイグレーション>>

include::lab-access.adoc[]

[#computerand]
== Compute Migration to a random node
メンテナンスやトラブルシューティングの際、VMを別のノードに移行する必要が生じることがあります。ライブマイグレーションとは、実行中の仮想マシンインスタンス（VMI）を中断することなく、別のノードに移動するプロセスのことです。

ifdef::vmware[]
.Rosetta Stone: VMware
****
[NOTE]
====
この機能は、VMwareの「Compute vMotion」機能、およびVM配置のための「Distributed Resource Scheduler (DRS)」に類似しています。
====
****
endif::[]

このラボでは、仮想マシンのコンピュートノード間でのライブマイグレーションを実行します。これを行うには、実行中のVMにアクセスし、現在のノードを確認した上で、別のノードに移行します。

[#computerandrequirements]
=== Requirements

* 基盤となる PVC が *ReadWriteMany (RWX)* アクセスモードを使用していること
* ポッドネットワークが bridge バインディングタイプで構成されていないこと
* ポート *49152 および 49153* がVMの virt-launcher ポッドで利用可能であること（これらのポートがマスカレードネットワークインターフェースで指定されている場合、ライブマイグレーションは失敗します）
* ソースノードとデスティネーションノードで CPU モデルが一致していること - Intel から AMD へのライブマイグレーションは実行できません

[#computerandinstructions]
=== Instructions

. *Webブラウザ* から <<labaccess,admin ユーザーとして OpenShift コンソールにログイン>>していることを確認し、次のステップに進みます。
+

. 左側のサイドパネルから、*Virtualization（仮想化） -> Virtual Machines（仮想マシン）* に移動します。*All projects（すべてのプロジェクト）* で *live-migrate* ネームスペースを選択し、*live-migrate-vm1* という名前の仮想マシンをクリックします。
+

. 仮想マシンは現在停止しています。青い *Play（起動）* ボタンをクリックして、仮想マシンを *Start（開始）* します。
+

image::exercise1/01-vm-start-001.png[title="Start the Virtual Machine", link=self, window=blank, width=100%]
+

. 仮想マシンのスケジュールが組まれ、起動し、以下のように *Running（実行中）* 状態になります。
+

image::exercise1/01-vm-start-002.png[title="Running Virtual Machine", link=self, window=blank, width=100%]
+

[NOTE]
====
仮想マシンが現在実行されているノードを確認するには、*VirtualMachine details（VirtualMachine の詳細）* ページの右側にある *General（一般）* ボックスの *Node（ノード）* 名を確認します。この例では、ノードは *worker-cluster-m2ssq-1* です。
====
+
image::exercise1/01-image-compute-002.png[title="Virtual machine details", link=self, window=blank, width=100%]
+

. 仮想マシンを別のノードに移行するには、*Actions（アクション）* メメニューをクリックし、*Migration（移行）* にカーソルを合わせてから、*Compute（コンピュート）* をクリックします。
+

image::exercise1/01-image-compute-003.png[title="Initiate migration", link=self, window=blank, width=100%]
+

. デフォルトの選択肢である、任意の *Node（ノード）* へ *Automatically（自動的）* に移行する設定のままにします。

+
image::exercise1/01-image-compute-004.png[title="Configure migration", link=self, window=blank, width=100%]

+
. *Migrate Virtual Machine（VirtualMachineの移行）* をクリックします。
+
. しばらく待つと、仮想マシンのステータスが *Running（実行中）* から *Migrating（マイグレーション中）* に変わります。

+
image::exercise1/01-image-compute-006.png[title="VM is migrating", link=self, window=blank, width=100%]

. その後さらに数秒すると、仮想マシンは *Running（実行中）* 状態に戻ります。
+
[NOTE]
====
*VirtualMachine details（VirtualMachine の詳細）* ページの右側にある *General（一般）* ボックスをもう一度見ると、ノードが変更されていることが確認できます。
====
+
image::exercise1/01-image-compute-007.png[title="VM is running", link=self, window=blank, width=100%]


. VMは実行状態を維持したまま、新しいワーカーノードに移行されました。

[#compute]
== Compute Migration to a specific node

メンテナンスやトラブルシューティングの際、仮想マシンをクラスター内の *特定（specific）* のノードに移行する必要がある場合があります。

[NOTE]
====
このタイプのマイグレーションの要件は、前のレッスンと同じです。
====

[#computeinstructions]
=== Instructions

. *Webブラウザ* から <<labaccess,admin ユーザーとして OpenShift コンソールにログイン>>していることを確認し、次のステップに進みます。

. 左側のサイドパネルから、*Virtualization（仮想化） -> Virtual Machines（仮想マシン）* に移動します。

. *All projects（すべてのプロジェクト）* で *live-migrate* ネームスペースを選択し、*live-migrate-vm1* という名前の仮想マシンをクリックします。

. 仮想マシンは前のレッスンから引き続き *Running（実行中）* 状態になっているはずです。もしそうでない場合は、青い *Play（起動）* ボタンをクリックして仮想マシンを *Start（開始）* し、*Running（実行中）* 状態になるまで待ちます。
+
image::exercise1/01-image-compute-002.png[title="Virtual machine details", link=self, window=blank, width=100%]

+
. 仮想マシンを別のノードに移行するには、*Actions（アクション）* メメニューをクリックし、*Migration（移行）* にカーソルを合わせてから、*Compute（コンピュート）* をクリックします。

+
image::exercise1/01-image-compute-003.png[title="Initiate migration", link=self, window=blank, width=100%]

+
. *Select the target Node to migrate your VirtualMachine to（VirtualMachineを移行するターゲットノードを選択）* ウィンドウが表示されます。

image::exercise1/01-image-compute-004.png[title="Configure migration", link=self, window=blank, width=100%]

. *Specific Node（特定のノード）* を選択すると、利用可能なノードのリストが表示されます。いずれかのワーカーノードの横にあるチェックボックスを選択し、*Migrate Virtual Machine（VirtualMachineの移行）* をクリックします。

+
image::exercise1/01-image-compute-005.png[title="Select worker node", link=self, window=blank, width=100%]

image::exercise1/01-image-compute-006.png[title="VM is migrating", link=self, window=blank, width=100%]

+
. その後しばらくすると、仮想マシンのステータスは *Running（実行中）* 状態に戻ります。

+
[NOTE]
====
*VirtualMachine details（VirtualMachine の詳細）* ページの右側にある *General（一般）* ボックスをもう一度確認すると、ノードが変更されており、特にリストから選択したノードに変わっていることがわかります。
====

+
image::exercise1/01-image-compute-007.png[title="VM is running", link=self, window=blank, width=100%]

[#storage]
== Storage Migration
ストレージ環境には独自のライフサイクルやパフォーマンスプロファイルがあることが多いため、ライフサイクル管理やパフォーマンス管理のために仮想マシンをストレージデバイス間で移動できることが重要です。OpenShift Virtualizationでは、仮想マシンを実行したまま、PVCをある StorageClass から別の StorageClass へ移行することができます。

ifdef::vmware[]
.Rosetta Stone: VMware
****
[NOTE]
====

この機能は、VMwareの「Storage vMotion」機能に類似しています。

これには、ディスク配置のための「Storage Distributed Resource Scheduler (SDRS)」も含まれます。
====
****
endif::[]

[#storagerequirements]
=== Requirements

* ストレージクラスの移行には、*Migration Toolkit for Containers (MTC)* オペレーターがインストールされている必要があります。

[NOTE]
====
MTC は非推奨（deprecated）となっており、エンジニアリングチームはストレージクラスの移行における *MTC* の要件を削除する作業を進めています。

この機能は *4.21* をターゲットにしており、エンジニアリングチームは *4.20.z* へのバックポートを計画しています。

詳細については、 https://issues.redhat.com/browse/VIRTSTRAT-97[VM Storage Migration]をご確認ください。
====

[#storageinstructions]
=== Instructions

. *admin* ユーザーとして OpenShift コンソールにログインしていることを確認し、次のステップに進みます。

+
. 左側のサイドパネルから、*Virtualization（仮想化） -> Virtual Machines（仮想マシン）* に移動します。

+
*All projects（すべてのプロジェクト）* で *live-migrate* ネームスペースを選択し、*live-migrate-vm1* という名前の仮想マシンをクリックします。
+
image::exercise1/01-image-storage-001.png[title="Virtual machine details", link=self, window=blank, width=100%]

+
. *Configuration（設定）* タブをクリックし、*Storage（ストレージ）* を選択します。仮想マシンには現在、*ocs-external-storagecluster-ceph-rbd* という名前の *Storage class（ストレージクラス）* を使用している 30GB のブート可能ディスクが1つだけ存在していることが確認できます。

+
image::exercise1/01-image-storage-002.png[title="View storage details", link=self, window=blank, width=100%]
+
. *Console（コンソール）* タブをクリックします。必要に応じて、青い *Connect（接続）* ボタンをクリックします。

+
コンソールウィンドウの上部に表示されている *User name（ユーザー名）* と *Password（パスワード）* の資格情報を使用し、*Copy to clipboard（クリップボードにコピー）* と *Paste to console（コンソールに貼り付け）* を使って仮想マシンにログインします。
+
. この演習の残りの作業を実行している間も動き続けるように、`top` や `ping` などのコマンドを実行しておきます。
+
image::exercise1/01-image-storage-003.png[title="Run command on console", link=self, window=blank, width=100%]
+
. *Actions（アクション）* メニューをクリックし、*Migration（移行）* にカーソルを合わせてから、*Storage（ストレージ）* をクリックします。
+
image::exercise1/01-image-storage-004.png[title="Initiate storage migration", link=self, window=blank, width=100%]
+
. *Migrate VirtualMachine storage（VirtualMachine ストレージの移行）* ウィンドウでオプションを確認し、デフォルトのまま *Next（次へ）* をクリックします。
+
image::exercise1/01-image-storage-005.png[title="Migration details", link=self, window=blank, width=100%]
+
. *Destination StorageClass（移行先 StorageClass）* として、*custom-storage-class* という名前の新しい *StorageClass* を選択し、*Next（次へ）* をクリックします。
+
image::exercise1/01-image-storage-006.png[title="Select destination StorageClass", link=self, window=blank, width=100%]
+
. 最終的なストレージマイグレーションの構成を確認し、*Migrate VirtualMachine storage（VirtualMachineストレージの移行）* をクリックします。
+
image::exercise1/01-image-storage-007.png[title="Review migration details", link=self, window=blank, width=100%]

. 次のウィンドウにストレージマイグレーションのステータスが表示されます。右上隅の *X* をクリックしてこのウィンドウを閉じても安全です。または、*View storage migrations（ストレージ移行の表示）* のリンクをクリックしても構いません。*Stop（停止）* ボタンはクリックしないでください。
+
image::exercise1/01-image-storage-008.png[title="Migration has started", link=self, window=blank, width=100%]

. ストレージマイグレーションのステータスを確認するには、左側のサイドパネルから *Migration for Virtualization（仮想化のマイグレーション）* に移動し、*Storage migrations（ストレージマイグレーション）* をクリックします。マイグレーションプランは *Pending（保留中）* 状態から始まり、*Running（実行中）* へ遷移し、最後に *Completed（完了）* になります。

+
[NOTE]
====
このラボ環境では、マイグレーションが開始されるまでに1分以上、完了するまでに5分以上かかる場合があります。
====
+
image::exercise1/01-image-storage-009.png[title="Migration is running", link=self, window=blank, width=100%]

. マイグレーションが *Complete（完了）* したら、左側のサイドパネルから再度 *Virtualization（仮想化） -> Virtual Machines（仮想マシン）* に移動します。

+
*All projects（すべてのプロジェクト）* で *live-migrate* ネームスペースを選択し、*live-migrate-vm1* という名前の仮想マシンをクリックします。
+
*Configuration（設定）* タブをクリックし、*Storage（ストレージ）* を選択します。*Source（ソース）* の名前が変更され、*Storage class（ストレージクラス）* がストレージマイグレーションで選択した *custom-storage-class* になっていることが確認できます。
+
image::exercise1/01-image-storage-010.png[title="VM has been migrated", link=self, window=blank, width=100%]

+
. *Console（コンソール）* タブに戻ると、ping がまだ実行中であり、マイグレーション中も中断されなかったことが確認できます。

+
image::exercise1/01-image-storage-011.png[title="No interruption during storage migration", link=self, window=blank, width=100%]

[IMPORTANT]
====
次のラボ用に十分なリソースを確保するため、*Terminal（ターミナル）* ウィンドウから *virtctl* を使用して *live-migrate-vm1* VM を停止してください。

[,shell,role="execute wrap",subs="attributes"]
----
virtctl stop live-migrate-vm1 -n live-migrate
----

*VM live-migrate-vm1 was scheduled to stop*
