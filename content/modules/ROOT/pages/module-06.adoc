
= Fencing and How to Handle Node Failure

ifdef::vmware[]
.Rosetta Stone: VMware
****
フェンシング（Fencing）および自己ノードリメディエーション（Self Node Remediation）は、VMwareのインフラストラクチャHA（Infrastructure HA）およびノード障害（Node Failure）機能に類似しています。
****
endif::[]

フェンシング（Fencing）は、クラスターリソースを保護し、データの整合性を確保するための極めて重要なメカニズムです。これは、応答しないノードや障害の発生したノードを隔離することによって機能します。このプロセスは、複数のノードが共有ストレージに同時に書き込もうとすることで発生する「スプリットブレイン（Split-brain）」シナリオを防ぎ、データの破損を確実に回避するために不可欠です。

このラボでは、*Node Health Check Operator* と *Self Node Remediation Operator* の機能を実証します。ノードの障害をシミュレートし、仮想マシンのマイグレーションプロセスとその後のノードの回復を観察します。


[IMPORTANT]
====
このラボでは *Node Health Check (NHC)* および *Self Node Remediation (SNR)* オペレーターを介したフェンシングの実演を行いますが、現場への推奨事項としては、ワークロードの回復時間が最も短い *Fence Agents Remediation* の採用を顧客に強く促すことです。

NHC と SNR は、管理インターフェースが存在しない場合や、クラスターから BMC にアクセスできない状況でのみ推奨されます。

この領域の機能は現在も進化を続けています。開発チームは、回復時間を60秒未満に短縮するための追加の機能拡張に取り組んでいます。

開発の進捗状況については、 https://issues.redhat.com/browse/VIRTSTRAT-77[Fencing: Additional out-of-band health checks for faster remediation] をフォローして確認することができます。
====

[#fencinginstructions]
== Instructions

. *Webブラウザ* からOpenShiftコンソールに、画面右側の *ターミナル* ウィンドウからCLIに、それぞれ *admin* ユーザーとしてログインしていることを確認し、次のステップに進みます。
+

. OpenShiftコンソールを使用し、左側のサイドパネルから *Compute（コンピュート） → NodeHealthChecks（ノードヘルスチェック）* に移動します。
+
*Create NodeHealthCheck（NodeHealthCheckの作成）* ボタンをクリックします。

+
image::exercise6/06-image-nodehealthcheck-v2.png[title="Create NodeHealthCheck", link=self, window=blank, width=100%]
+
. *NodeHealthCheck* の名前を `workerhealthcheck` にします。
+
*Remediation template（リメディエーションテンプレート）* は *Self node remediation（自己ノードリメディエーション）* のままにします。
+
*Selector labels（セレクターラベル）* のドロップダウンから、*Role* の下にある *Worker* を選択します。*Labels（ラベル）* を選択する必要はありません。

+
image::exercise6/06-image-nodehealthcheck2.png[title="Create NodeHealthCheck", link=self, window=blank, width=100%]

+
同じ画面のさらに下部で、*Min healthy（最小健全数）*、ならびに *Type（タイプ）* と *Status（ステータス）* の両方のドロップダウンはデフォルトのままにします。
+
2つの *Unhealthy Conditions（不健全な条件）* の期間（Duration）を両方とも *5s* に変更します。
+
image::exercise6/06-image-nodehealthcheck3.png[title="Create NodeHealthCheck", link=self, window=blank, width=100%]

+
*Create（作成）* をクリックします。

+
. 次に、*fencing-vm1* が実行されているノードを見つけます。この例では、ノードは *worker-cluster-r2k68-1* です。

+
[source,text,role=execute]
.Get VM info
----
oc get vmi fencing-vm1 -n fencing
----
+
[source,console]
.Output
----
NAME          AGE    PHASE     IP            NODENAME                 READY
fencing-vm1   169m   Running   10.235.0.29   worker-cluster-r2k68-1   True
----
+
より正確に実行するため、*jq* を使用して *nodeName* のみを返し、後で使用できるように変数に割り当てます。

+
[source,text,role=execute]
.Get VM info into a variable
----
node=`oc get vmi fencing-vm1 -n fencing -o json | jq -r '.status.nodeName'`
----
+
[source,text,role=execute]
.Show node name
----
echo ${node}
----
+
[source,console]
.Output
----
worker-cluster-r2k68-1
----

+
. 障害を発生させる前に、2つ目のターミナルでノードの状態を監視するウォッチ（Watch）を開始し、ワーカーノードがリメディエーション（修復）される様子を確認できるようにします。

+
[source,text,role=execute]
.Monitor the worker nodes
----
oc get nodes -w
----
+
. 障害条件を作成します。ステップ4のノードに対し、*${node}* 変数と *oc debug* を使用してアクセスし、*kubelet* サービスを停止してノードを不健全な状態にします。

+
[source,text,role=execute]
.Debug into the node
----
oc debug node/${node}
----
+
[source,text,role=execute]
.chroot
----
chroot /host
----

+
[source,text,role=execute]
.Force the node into a unhealthy state
----
systemctl stop kubelet
----

+
[source,console]
.Output
----
Removing debug pod ...
----

+
. *debug ポッド* が終了したら、仮想マシン *fencing-vm1* のステータスを監視します。自己リメディエーションプロセスによってノードが隔離され、ワークロードがマイグレーションされるのを確認できます。

+
[source,text,role=execute]
.Get VM info
----
oc get vmi fencing-vm1 -n fencing -w
----

+
. OpenShiftコンソールを使用し、左側のサイドパネルから *Ecosystem（エコシステム） → Installed Operators（インストール済みのオペレーター） → Self Node Remediation Operator → Self Node Remediation* に移動します。
+
[NOTE]
====
ノードが *NotReady* 状態になるまで *SelfNodeRemediation* オペランドは表示されません。これには数秒かかります。
====

+
ここでは、自己リメディエーションオブジェクトを確認し、ノードに対する処理の進捗を追うことができます。

+
image::exercise6/06-image-nodehealthcheck4.png[title="Node remediation running", link=self, window=blank, width=100%]

+
. 2つのターミナルウィンドウから、以下の挙動が確認できます。
+
.. ワーカーノードがリメディエーションされ、*Ready* → *NotReady* → *NotReady,SchedulingDisabled* → *Ready,SchedulingDisabled* → *Ready* と遷移していく様子。

+
[source,console]
----
NAME                     STATUS                        ROLES    AGE     VERSION
worker-cluster-r2k68-1   Ready                         worker   4h15m   v1.33.5
worker-cluster-r2k68-1   Ready                         worker   4h16m   v1.33.5
worker-cluster-r2k68-1   NotReady                      worker   4h16m   v1.33.5
worker-cluster-r2k68-1   NotReady                      worker   4h16m   v1.33.5
worker-cluster-r2k68-1   NotReady,SchedulingDisabled   worker   4h16m   v1.33.5
worker-cluster-r2k68-1   NotReady,SchedulingDisabled   worker   4h16m   v1.33.5
worker-cluster-r2k68-1   Ready,SchedulingDisabled      worker   4h17m   v1.33.5
worker-cluster-r2k68-1   Ready,SchedulingDisabled      worker   4h17m   v1.33.5
worker-cluster-r2k68-1   Ready,SchedulingDisabled      worker   4h17m   v1.33.5
worker-cluster-r2k68-1   Ready                         worker   4h18m   v1.33.5
worker-cluster-r2k68-1   Ready                         worker   4h18m   v1.33.5
----
+
.. ノードが故障したときにVMが *Running* → *Failed* 状態になり、続いてリメディエーションが行われて新しいノードに移行されるにつれて *Pending* → *Scheduling* → *Scheduled* → *Running* へと遷移していく様子。

+
[source,console]
----
NAME          AGE   PHASE        IP            NODENAME                 READY
fencing-vm1   36m   Running      10.235.0.14   worker-cluster-r2k68-1   True
fencing-vm1   37m   Running      10.235.0.14   worker-cluster-r2k68-1   False
fencing-vm1   38m   Failed       10.235.0.14   worker-cluster-r2k68-1   False
fencing-vm1   38m   Failed       10.235.0.14   worker-cluster-r2k68-1   False
fencing-vm1   0s    Pending
fencing-vm1   1s    Scheduling                                          False
fencing-vm1   10s   Scheduled                  worker-cluster-r2k68-3   False
fencing-vm1   10s   Running      10.232.2.41   worker-cluster-r2k68-3   True
----

[IMPORTANT]
====
次のラボ用に十分なリソースを確保するため、*Terminal（ターミナル）* ウィンドウから *virtctl* を使用して *fencing-vm1* VM を停止してください。

[,shell,role="execute wrap",subs="attributes"]
----
virtctl stop fencing-vm1 -n fencing
----
*VM fencing-vm1 was scheduled to stop*
