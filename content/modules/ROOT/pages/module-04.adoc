
= Affinity and Anti-Affinity for VM Placement

ifdef::vmware[]
.Rosetta Stone: VMware
****
OpenShiftのアフィニティルール（Affinity Rules）は、VMwareの仮想マシンアフィニティルールに類似しています。
さらに、VMのスマートで自動化された再スケジュールを行うために OpenShift Descheduler を採用した場合、この機能は VMware の VMアフィニティルールと Distributed Resource Scheduler (DRS) を組み合わせたものに類似することになります。
ただし、それについてはこのモジュールでは扱いません。
****
endif::[]

このラボでは、仮想マシンに対してノードアフィニティ（Node Affinity）、ポッドアフィニティ（Pod Affinity）、およびポッドアンチアフィニティ（Pod Anti-Affinity）を使用および適用する方法を紹介します。このラボでは、現実世界のシナリオにおけるこれらの原則の具体的な適用に焦点を当て、それらがどのように機能するかについての技術的な理解を深めます。

[#nodeaffinity]
== Node Affinity

ノードアフィニティ（Node Affinity）は、特定のノードまたはノードのグループに仮想マシンを引き付けるようスケジューラーに指示する一連のルールです。これらのルールは、ノードに適用されているラベルの一致に依存します。

ノードアフィニティの主なユースケースは、ノード上の対応するラベルを一致させることで、特定のGPUモデルや大容量のRAMなど、特定の機能や要件を持つノード上でのみVMが実行されるようにすることです。

このラボでは、ノードアフィニティがどのように設定され、どのように機能するかを実証します。

[#nodeaffinityinstructions]
=== Instructions

. *Webブラウザ* からOpenShiftコンソールに、画面右側の *ターミナル* ウィンドウからCLIに、それぞれ *admin* ユーザーとしてログインしていることを確認し、次のステップに進みます。
+

. *node-affinity-vm* 仮想マシンを起動します。

+
[.wrap,console,role=execute]
----

----
+
[source,console]
----

----

. *VirtualMachineInstance* が実行中であることを確認し、どのノードで動作しているかを確認します。*VirtualMachineInstance* がまだスケジュール（Scheduling）フェーズにある場合は、コマンドを再実行してください。
+
[.wrap,console,role=execute]
----
oc get vmi -n affinity
----
+
出力は以下のようになります（*IP* や *NODENAME* は環境によって異なります）。
+
[source,console]
----

----

+
. *node-affinity-vm* が現在実行されて *いない* いずれかのノードに、`zone=east` というラベルを設定します。
+
次のコマンドは、*node-affinity-vm* VMが実行されていないノードの名前を1つ返し、それにラベルを付与します。
+
... すべてのノードのリストを取得します（*oc get nodes -o name*）
+
... 反転マッチ（除外検索）を行って一致しない行を選択し、1つの結果を返します（*grep -v -m 1*）

... *node-affinity-vm* VMが現在実行されている *nodeName* を指定します（*oc get vmi node-affinity-vm -n affinity -o json | jq -r '.status.nodeName'*）

+
[source,text,role=execute]
.Set the label
----

----

+
[source,console]

----

----

+
あるいは、OpenShiftコンソールを使用し、左側のサイドパネルから *Compute（コンピュート） → Nodes（ノード）* に移動し、*node-affinity-vm* が実行されていないノードを選択し、3つのドットをクリックして *Edit labels（ラベルの編集）* を選択します。

+
. ラベルが適用されたことを確認します。

+
[source,text,role=execute]

----

+
あるいは、OpenShiftコンソールを使用し、左側のサイドパネルから *Compute（コンピュート） → Nodes（ノード）* に移動し、ラベルを追加したノードを選択して *Details（詳細）* タブをクリックし、*Labels（ラベル）* セクションに *zone=east* があることを確認します。

image::exercise4/04-image-affinity-check-labels.png[title="Check the Label", link=self, window=blank, width=100%]

+
*All projects（すべてのプロジェクト）* の下で、*affinity* ネームスペースを選択し、*node-affinity-vm* という名前の仮想マシンをクリックします。

image::exercise4/04-image-affinity01.png[title="Node Affinity Navigation", link=self, window=blank, width=100%]

[#podaffinity]
== Pod Affinity
ポッドアフィニティ（Pod Affinity）は、特定のラベルを持つVM（またはポッド）を同じノード上にコロケーション（同配置）させるスケジューリングルールです。

ポッドアフィニティを使用する主なメリットは、低遅延通信を必要とする依存関係のあるVMやサービスを確実に同じノードに配置することで、パフォーマンスを向上させられる点にあります。

このラボでは、ポッドアフィニティの設定と使用方法を実証します。

[#podaffinityinstructions]
=== Instructions

. *Webブラウザ* からOpenShiftコンソールに、画面右側の *ターミナル* ウィンドウからCLIに、それぞれ *admin* ユーザーとしてログインしていることを確認し、次のステップに進みます。
+

. まず、前のラボの既存の *node-affinity-vm* を活用します。これが、アフィニティを実証するために使用する2つのVMのうちの1つとして機能します。
+
ラベルを追加する方法は2つあります。
+
.. CLIまたはコンソールからVMのYAMLを編集する
+
.. *oc label* を使用する
+
どちらの方法にもメリットとデメリットがあります。多くの本番環境のユースケースでは、これらが組み合わせて使用されます。
+
[loweralpha, start=1]
... VMのYAMLを編集する場合、新しいラベルを有効にするためにVMの再起動が必要です。これは、VM上のラベルが *virt-launcher* ポッドに引き渡される必要があり、それが再起動後にしか行われないためです。
+
... *oc label* の使用は一時的（エフェメラル）であり、VMの再起動後に失われます。これは、ラベルが *virt-launcher* ポッドに直接適用されて即座に効果を発揮しますが、VMオブジェクト自体には設定されないためです。

+
. 設定を永続的なものにするため、OpenShift CLIを使用して *node-affinity-vm* に `app: fedora` というラベルを設定します。
+
ラベルは、YAMLの最上部にある *metadata.labels* ではなく、*spec.template.metadata.labels* の下に追加してください。

+
[source,text,role=execute]
.Modify the VM
----

----

+

----

  instancetype:
    kind: virtualmachineclusterinstancetype

  runStrategy: Manual
  template:
    metadata:
      creationTimestamp: null

        nodeAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - preference:
              matchExpressions:
              - key: zone
                operator: In
                values:
                - east
            weight: 75
      architecture: amd64
      domain:
        devices:
          autoattachPodInterface: false
          disks:
          - disk:
              bus: virtio
            name: cloudinitdisk
          interfaces:
          - macAddress: 02:f9:4a:ad:c3:06
            masquerade: {}
            name: default
        firmware:
          serial: 212f42c0-c2ac-4581-96e2-5297e2436a8d
          uuid: ad3a3d79-9539-4585-826e-9cca5a71f775
        machine:
          type: pc-q35-rhel9.6.0
        resources: {}
      networks:
      - name: default
        pod: {}
      subdomain: headless
      volumes:
      - dataVolume:
          name: node-affinity-vm-volume
        name: rootdisk
      - cloudInitNoCloud:
          userData: |
            chpasswd:
              expire: false
            password: redhat
            user: rhel
        name: cloudinitdisk
----

.. まず、*pod-affinity-vm* 仮想マシンを起動します。

[#podantiaffinity]
== Pod Anti-Affinity

ポッドアンチアフィニティ（Pod Anti-Affinity）は、ワークロードに対してクラスターをまたぐ高可用性（HA）を実現するための極めて重要な機能です。これは、関連するVMを同じノードに共存させないようスケジューラーに指示することで機能します。

主なメリットは、アプリケーションの復元力（レジリエンス）を高める点にあります。同じサービス（データベースクラスターなど）に属するVMが異なるノードに分散して配置されるようになります。これにより、単一のノードに障害が発生しても、サービス全体の停止を引き起こすことはなくなります。

このラボでは、ポッドアンチアフィニティの設定と使用方法を実証します。

[#podantiaffinityinstructions]
=== Instructions

. *Webブラウザ* からOpenShiftコンソールに、画面右側の *ターミナル* ウィンドウからCLIに、それぞれ *admin* ユーザーとしてログインしていることを確認し、次のステップに進みます。
+

. まず、前のラボの既存の *node-affinity-vm* と *app: fedora* ラベルを活用します。これが、アンチアフィニティを実証するために使用する2つのVMのうちの1つとして機能します。
+
. 次に、*pod-anti-affinity-vm* の構成を行います。
+
.. まず、*pod-anti-affinity-vm* 仮想マシンを起動します。

----

+
*Configuration（設定）* タブから *Scheduling（スケジューリング）* を選択し、*Affinity rules（アフィニティルール）* の下にある *青い鉛筆* アイコンをクリックして新しいルールを追加します。

image::exercise4/04-image-affinity06.png[title="Add Pod Anti-Affinity Rule", link=self, window=blank, width=100%]

+
. 表示された新しいウィンドウから、*Add affinity rule（アフィニティルールの追加）* をクリックします。
+
. *Type（タイプ）* を *Workload (pod) Anti-Affinity（ワークロード（ポッド）アンチアフィニティ）* に変更します。
+
. *Condition（条件）* は *Required during scheduling（スケジューリング時に必須）* のままにし、*Topology key（トポロジーキー）* もデフォルト値のままにします。
+
. *Workload labels（ワークロードラベル）* の下にある *Add expression（式の追加）* をクリックします。
+
*Key（キー）* フィールドに `app`、*Values（値）* フィールドに `fedora` を設定し、*Add（追加）* をクリックします。
+
これは、*node-affinity-vm* に適用されているものと同じラベルです。
+
最終的なポッドアフィニティルールは下の画像のようになります。
+
これを確認し、*Save affinity rule（アフィニティルールの保存）* をクリックします。
+
image::exercise4/04-image-affinity07.png[title="Add Pod Anti-Affinity Rule", link=self, window=blank, width=100%]

+
. *Apply Rules（ルールの適用）* をクリックします。
+
. それでは、*pod-anti-affinity-vm* のポッドアフィニティルールを確認してみましょう。
+
この情報は、GUIで *pod-anti-affinity-vm* の *YAML* を検査するか、CLIを使用して確認できます。

+
[source,text,role=execute]
----
oc get vm pod-anti-affinity-vm -n affinity -o jsonpath='{.spec.template.spec.affinity}{"\n"}'
----
+
[source,console]
.Output
----
{"podAntiAffinity":{"requiredDuringSchedulingIgnoredDuringExecution":[{"labelSelector":{"matchExpressions":[{"key":"app","operator":"In","values":["fedora"]}]},"topologyKey":"kubernetes.io/hostname"}]}}
----

+
. OpenShiftコンソールを確認するか、OpenShift CLIを使用して、*pod-anti-affinity-vm* と *node-affinity-vm* がどこで実行されているかを確認します。

+
[source,text,role=execute]
----
oc get vmi pod-anti-affinity-vm node-affinity-vm -n affinity
----
+
[source,console]
.Output
----
NAME                   AGE   PHASE     IP            NODENAME                 READY
pod-anti-affinity-vm   30m   Running   10.233.0.44   worker-cluster-dvddt-1   True
node-affinity-vm        5m   Running   10.233.0.46   worker-cluster-dvddt-1   True
----

+
[NOTE]
====
クラスターの負荷が低いため、*pod-anti-affinity-vm* と *node-affinity-vm* がすでに異なるノードに配置されている可能性があります。そうだとしても、ポッドを何度再起動しても、それらが同じノードになることは決してありません。
====

. 先ほど述べたように、外部からの力によってVMが移動または再起動されない限り、新しいアフィニティルールは効果を発揮しません。
+
手動で変更を適用するには、VMをライブマイグレーションするか再起動します。自動的に強制適用させるには、*Descheduler* に *AffinityAndTaints* プロファイルを構成します。

+
. *pod-anti-affinity-vm* VMを再起動します。

+
[.wrap,console,role=execute]
----
virtctl restart pod-anti-affinity-vm -n affinity
----

+
[source,console]
----
VM pod-anti-affinity-vm was scheduled to restart
----

+
. VMが再起動すると、ポッドアンチアフィニティが有効になります。
+
これは、GUIで *pod-anti-affinity-vm* および *node-affinity-vm* に移動し、*VirtualMachine details（VirtualMachineの詳細）* ページの右側にある *General（一般）* ボックスの *Node（ノード）* 名を確認するか、OCP CLIを使用して検証できます。

+
[source,text,role=execute]
----
oc get vmi pod-anti-affinity-vm node-affinity-vm -n affinity
----

+
[source,console]
.Output
----
NAME                   AGE   PHASE     IP            NODENAME                        READY
pod-anti-affinity-vm   43s   Running                 control-plane-cluster-dvddt-1   True
node-affinity-vm       58m   Running   10.233.0.35   worker-cluster-dvddt-1          True
----

+
. ポッドアンチアフィニティルールによって、*pod-anti-affinity-vm* が *node-affinity-vm* とは異なるノードで実行されていることが確認できます。

[IMPORTANT]
====
次のラボ用に十分なリソースを確保するため、*Terminal（ターミナル）* ウィンドウから *virtctl* を使用して、*node-affinity-vm*、*pod-affinity-vm*、および *pod-anti-affinity-vm* VM を停止してください。


[,shell,role="execute wrap",subs="attributes"]
----
virtctl stop node-affinity-vm -n affinity
----

*VM node-affinity-vm was scheduled to stop*

[,shell,role="execute wrap",subs="attributes"]
----
virtctl stop pod-affinity-vm -n affinity
----

*VM pod-affinity-vm was scheduled to stop*

[,shell,role="execute wrap",subs="attributes"]
----
virtctl stop pod-anti-affinity-vm -n affinity
----

*VM pod-anti-affinity-vm was scheduled to stop*
====
